# Credential Lifecycle Contract

This contract applies to API keys, OAuth/OIDC client credentials, signing and
TLS keys, ACME account keys, database/cache credentials, service tokens, and
authenticated diagnostic artifacts.

## States and required evidence

| State | Required control | Value-free evidence |
|---|---|---|
| Requested | Named owner, purpose, scope, environment, expiry/rotation policy | Ticket/change ID |
| Issued | Generated by the authoritative provider or approved secret store | Credential identifier/class only |
| Stored | Encrypted secret store or root/service-owned runtime file | Store path/record name, not value |
| Injected | Runtime-only, least-privilege consumer, no image/Git persistence | Service and variable/field name |
| Active | Health and authentication verified without dumping environment | Verification timestamp and result |
| Rotating | New credential validated before old credential is revoked where overlap is supported | Old/new logical IDs and status only |
| Revoked | Provider-side invalidation confirmed | Revocation timestamp/result |
| Retired | Runtime references, caches, backups, CI artifacts, and documentation cleaned | Checklist and approvals |

Credential values, private-key fingerprints, JWTs, session IDs, authorization
headers, and connection strings are never lifecycle evidence.

## Ownership and separation

- The service owner approves scope and validates application behavior.
- A credential administrator performs issue/rotate/revoke actions in the
  authoritative provider or secret store.
- A reviewer confirms least privilege, repository cleanliness, and value-free
  evidence.
- Git history rewriting and force-updating shared refs require separate owner
  approval; rotation does not imply that approval.

One person may hold multiple roles only when the emergency change record names
that exception. The scanner never decides whether a credential is safe to keep.

## Creation and injection

- Generate credentials outside the repository and shell history.
- Prefer short-lived workload identity over static credentials where supported.
- Grant only the environment, tenant, API, database, and operation scope needed.
- Store runtime files with directory mode `0700` and file mode `0600` unless a
  narrower service-specific ownership model is documented.
- Inject values without `env`, `set -x`, command-line arguments visible in
  process lists, or templated logs.
- Do not reuse one credential across production, staging, CI, and local use.

## Rotation triggers

Rotate and revoke when a value or private artifact is committed, logged,
included in a diagnostic capture, stored with unintended read permissions,
copied outside the approved store, exposed to an untrusted actor, or when its
owner/scope is no longer valid. Suspected exposure is sufficient; reading the
value to prove exposure is not required.

Production promotion remains frozen for an affected credential until the
service owner confirms the replacement works and the old credential is revoked
or otherwise rendered unusable.

## Retirement

Retirement removes runtime references, local rollback copies, backup copies,
CI artifacts, caches, and stale documentation. Git-history remediation follows
only after rotation because deleting a blob does not invalidate a credential.
Rollback restores application configuration or code, never a revoked or exposed
credential.
