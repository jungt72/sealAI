# Audit: Frontend Routing & Strapi

## Findings
- **A /dashboard → /chat routing exists** – Both the deployed container and the repo contain the same `middleware` logic that intercepts `/dashboard` requests, checks for `chat_id`, and redirects via `/chat/${chat_id}` plus the server component in `frontend/src/app/dashboard/page.tsx` (see `docker exec frontend sed -n "1,140p" /app/middleware.ts` and `frontend/middleware.ts`). This means the routing change is implemented in code, but we can only verify it if that code is actually deployed.
- **B running frontend still uses the GHCR image so repo changes are stale** – `docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}\t{{.Ports}}"` shows `frontend` running `ghcr.io/jungt72/sealai-frontend:latest`; `docker compose config` still advertises the same image even though a `build` block exists in `docker-compose.yml`. Because `sealai` has never been rebuilt from the workspace, every edit in this repo (including the routing change above) sits beside the running container. As a consequence the container’s `healthcheck` keeps failing with `/bin/sh: curl: not found` (`docker inspect frontend --format '{{json .State.Health}}'`), so the service remains `unhealthy` and is not fulfilling requests reliably.
- **C Strapi is only reachable on the internal network** – The `biz` compose project hosts Strapi in `docker-compose.biz.yml` without any published ports; the service only joins the `sealai_default` network, so it is not reachable via `sealai.net`. Nginx’s active config (`nginx/default.conf`) contains no `/admin`, `/uploads` or `/api` proxies for Strapi (the backup file still shows those blocks, but the live file only proxies `/api/v1/` toward the backend and everything else to the frontend). As a result, `curl -sS -I https://sealai.net/api/hero?populate=*` and the dashboard URLs all fail with `curl: (6) Could not resolve host: sealai.net`, while the frontend container can still reach Strapi via `http://strapi:1337` (`docker exec frontend wget -qSO- http://strapi:1337/api/hero?populate=*`). This mismatch means `NEXT_PUBLIC_STRAPI_URL` (set only to `http://strapi:1337` in `docker-compose.yml` and inside the container `printenv`) points to a host that browsers cannot resolve or reach, so public pages can never fetch Strapi data or images.

## Root Causes
- **A (routing)** – The code is correct, but the deployment pipeline still runs the immutable GHCR frontend image, so the latest middleware/page files never actually reach clients and cannot fulfill `/dashboard` → `/chat` redirects as intended.
- **B (frontend deployment + health)** – The compose stack relies solely on `ghcr.io/jungt72/sealai-frontend:latest` (`docker ps`/`docker compose config`) while the healthcheck defined in `docker-compose.yml` still expects `curl`; that binary is missing in the image (`docker inspect frontend`). The result is a perpetually `unhealthy` container that cannot recover or reload with the repo’s current build outputs.
- **C (Strapi reachability + env wiring)** – Strapi is isolated inside the `biz` compose project (`docker-compose.biz.yml`) and exposes no external port, while nginx lacks any proxy for Strapi endpoints (`nginx/default.conf` only proxies `/api/v1/` and `/api/chat`). The frontend relies on `NEXT_PUBLIC_STRAPI_URL`/`NEXT_PUBLIC_IMAGE_BASE_URL` being publicly reachable, but they only point to `http://strapi:1337` (see `docker exec frontend printenv`) which is a private Docker hostname that browsers cannot resolve (host curl to `sealai.net` fails). Consequently, Strapi-driven content never loads for real visitors.

## Minimal Fix Plan (Patchliste)
1. **`docker-compose.yml` (frontend service)** – remove the hard-coded `image` field (or point it to the locally built tag) so `docker compose up` rebuilds from `frontend/`. Keep the `build.args` for the `NEXT_PUBLIC_*` vars but ensure they default to the public-facing hostname (https://sealai.net). Also update the healthcheck to use `wget`/`node` or ensure the Dockerfile installs `curl` (`frontend/Dockerfile`) so the check reports success (`docker inspect frontend` shows curl missing).
2. **`frontend/Dockerfile`** – install `curl`/`wget` (or another lightweight HTTP client) before running the app so the healthcheck command defined in `docker-compose.yml` succeeds and the container becomes `healthy` again.
3. **`nginx/default.conf`** – add proxy blocks for the Strapi API, admin UI, uploads, and media so that `https://sealai.net/strapi`, `https://sealai.net/uploads`, or the JVM-proxied `/api/` URLs forward to `http://strapi:1337` (similar to `nginx/default.conf.backup`). Once nginx exposes Strapi endpoints, update the Compose env (and any `.env`/build args) so `NEXT_PUBLIC_STRAPI_URL`/`NEXT_PUBLIC_IMAGE_BASE_URL` use the public HTTPS URL (e.g., `https://sealai.net/strapi`), ensuring browsers fetch content from the reverse proxy rather than the Docker host name.

## Smoke Test Plan
1. `docker compose exec frontend wget -qSO- http://localhost:3000/dashboard?chat_id=test-123` → expect `HTTP/1.1 302 Found` with `Location: /chat/test-123` and a successful login redirect path.
2. `docker compose exec frontend wget -qSO- http://localhost:3000/chat` → expect `HTTP/1.1 200 OK` with the chat layout rendering (no 500s).
3. `curl -sS -I https://sealai.net/chat` → expect `HTTP/2 200` (or 301/302 to HTTPS-equivalent) and not a host resolution error.
4. `curl -sS -I https://sealai.net/api/hero?populate=*` → expect `HTTP/2 200` once nginx proxies to Strapi (previously produced `curl: (6)`); this proves `NEXT_PUBLIC_STRAPI_URL`/`IMAGE_BASE_URL` point to a reachable host.
5. `docker inspect frontend --format '{{json .State.Health}}'` → expect the JSON to report `"Status": "healthy"` and no recent `"ExitCode": 127` entries, proving the healthcheck now succeeds.
