# Live Parameter Form Sync Audit

## currentState origin
- `ChatContainer` keeps a `LanggraphState` (`currentState`) that mirrors what the backend SSE stream produces. Incoming `state_update` events (`onStateDelta`) call `mergeState`, which merges the delta into `currentState` while keeping existing fragments such as `messages` and `parameters`. Explicit resync paths (`refreshCurrentState`, `resyncStateFromServer`) also GET `/api/langgraph/state` and replace `currentState` with `_build_state_update_payload` (see `backend/app/api/v1/endpoints/langgraph_v2.py`).
- `_build_state_update_payload` emits `parameters` derived from the `SealAIState.parameters` Pydantic model (defined in `backend/app/langgraph_v2/state/sealai_state.py`). That model routes both `pressure_bar` and the legacy `pressure` alias into the same field (`Field(alias="pressure")`) and applies numeric coercion via `@field_validator("pressure_bar", mode="before")`, so the SSE payload always exposes canonical values even when the LLM surfaces legacy names.

## paramStore updates
- `useParamStore` exposes helpers such as `applyServerParams`, `applyRemoteDeltaFromSse`, `applyPatchAck`, and `applyLocalEdit`. When SSE or `/state` fetches deliver parameters, `ChatContainer` invokes `applyServerParameters`/`replaceParamFromServer`, which dispatch into `reduceParamStore` and eventually run `mergeServerParameters`/`computeAppliedKeys`. Those helpers honor the `dirty`/`pending` sets in the entry, meaning any key marked dirty is skipped until a patch ack clears it (unless the backend tags the key with `force_overwrite` via `parameter_meta`).
- Local edits are surfaced through `applyLocalEdit`, which writes the patch into the store, marks the field dirty/pending, and keeps it from being overwritten. Patches submitted through `patchAllParameters` call `/api/langgraph/parameters/patch`, which uses `apply_parameter_patch_lww` (`backend/app/langgraph_v2/utils/parameter_patch.py`) to enforce per-field versions, manage provenance, and publish `parameter_patch_ack` events. `applyParameterPatchAck` picks up the ack, normalizes aliases (`normalizeParameterPatch` maps `pressure` → `pressure_bar` only when the canonical key is absent), clears dirty/pending, and writes the acked values back to the store.

## form population
- `ParameterFormSidebar` tracks every field in `FORM_SECTIONS` as `fieldValues`. A `useEffect` watches `(currentState?.parameters || parameters)` and rebuilds `fieldValues` with `buildFieldValues`, so the sidebar will copy whatever the latest `currentState` or fallback store state exposes. The `parameters` prop comes directly from `useParamStore`, and `currentState` is passed from `ChatContainer`.
- `handleFieldChange` updates the local `fieldValues` immediately and forwards the sanitized value to `onFieldChange` (`ChatContainer`), which schedules `sendParameterPatch` with numeric debounce, so individual edits patch the backend without waiting for the submit button. The sticky submit area still calls `onSubmit`, which defers to `applyParametersWithChatMessage`/`patchAllParameters` for bulk updates or checkpoint flow, but the form UI itself already mutates the store via the field-change handler.
- Because the form owns `fieldValues` plus the param store data and keeps the submit button around for historical reasons, there are effectively two live copies: the local `fieldValues` and the `parameters` value the store tracks. SSE updates refresh `fieldValues` via `currentState` but can be ignored as soon as the store marks that key dirty.

## why latest write currently does not win
- `mergeServerParameters` (used by both `applyServerParams` and `applyRemoteDeltaFromSse`) explicitly skips copying keys that are still dirty, so a local edit that marks a key dirty forever blocks the next server update from overwriting that key until the dirty/pending flags are cleared by a patch ack. This is the “dirty blocks server” pattern causing older writes to stick around.
- Aliases are normalized through `normalizeParameterPatch`, but `pressure` only writes back to `pressure_bar` when `pressure_bar` is missing, so if the form already tracks `pressure_bar` and a chat patch arrives as `pressure`, the alias never overwrites. That is the “only update if empty” anti-pattern mentioned in the requirement.
- The dual sources—SSE’s `currentState.parameters` versus the param store’s `parameters` and the form’s `fieldValues`—widely drift once dirty gating kicks in, so the UI can display stale values while the SSE stream already has newer ones. Until `applyServerParams` always sets the store and the form trusts `currentState.parameters`, the last writer (chat or form) can lose because the other side is still flagged dirty on the client.
